# 用户修改审核 · Modification Review

> **作用**：用户在 AI 生成的 persona 基础上做任何修改时，**每次修改都必须经过可识别性审核**。这是进化模式的安全守门员——防止用户通过“事后修改”把一个安全的虚构 persona，悄悄改成近现代现实政治人物的近似克隆。
>
> **底线**：安全则允许；不安全则拒绝该具体设定，保留用户抽象意图，给出安全替代表达。

---

## 1. 什么时候触发

用户在 persona 创建后，表达“给他加个设定 / 改成 / 他应该是 / 把他的背景换成……”等**任何修改意图**时，本审核强制触发。修改可能针对任意一层（身份 / 人性 / 生活 / 政治 / 自我状态 / 内在冲突）。

---

## 2. 审核内容（逐项检查）

检查用户新增/修改的内容是否包含：

- 近现代现实政治人物姓名；
- 现实政党；
- 现实派系；
- 现实选举；
- 现实政策口号；
- 现实丑闻；
- 现实家庭关系；
- 现实任职轨迹；
- 现实遇刺 / 审判 / 政变 / 下台事件；
- 多个可识别信息组合后指向某个近现代现实政治人物。

> 单项通用特征（如“保守派”“曾任议员”）不触发；**独有**或**多项组合**才触发。判定标准与指纹清单见 `recognizability_review.md` 第 2 节。

---

## 3. 审核结果与处理

### 3.1 安全 → 允许修改

- 允许修改；
- 更新 persona（对应层）；
- 更新一致性检查（确保修改不与 persona 其他字段自相矛盾）。

### 3.2 不安全 → 拒绝该具体设定

- **拒绝该具体设定**（不是拒绝整个 persona，只拒绝不安全的那一项）；
- **保留用户的抽象意图**（提炼其想表达的政治类型/质感）；
- **给出安全替代表达**（删除指纹后的等效设定）。

---

## 4. 标准处理示例（落实 SPEC 第 9.2 节）

**用户修改**：

```text
给他加一个设定：父亲也是首相，自己提出三支经济政策，后来遇刺。
```

**处理**：

```text
该设定会高度指向近现代现实政治人物，不能加入。
可以改为：出身政治家族，曾提出一套经济复兴纲领，并因强势改革长期遭遇激烈反对。
```

拆解：

| 用户原设定 | 为什么不安全 | 安全替代 |
|---|---|---|
| 父亲也是首相 | 独有家庭轨迹（指纹） | 出身政治家族（抽象） |
| 提出三支经济政策 | 独有政策口号（指纹） | 曾提出一套经济复兴纲领（抽象） |
| 后来遇刺 | 独有结局方式（指纹） | 因强势改革长期遭遇激烈反对（抽象，去掉具体结局） |

> 处理时向用户透明说明：哪一项被拒、为什么、替代是什么。不悄悄改。

---

## 5. 修改分诊（判断改哪一层）

参考 colleague 的 correction 机制，先判断修改属于哪一层，再走审核：

| 用户修改意图 | 归属层 | 审核重点 |
|---|---|---|
| 姓名 / 国籍 / 党派 / 派系 / 选区 / 角色 | 身份层 / 政治职业层 | 政党派系专名、任职轨迹、现实人物 |
| 性格 / 弱点 / 欲望 / 恐惧 | 人性核心层 | 是否在强化对某现实人物的类比可识别性 |
| 家庭 / 私人关系 / 经历 | 生活质感层 | 现实家庭关系、丑闻、成形事件指纹 |
| 意识形态坐标 / 政策主张 | 政治职业层 | 现实政策口号、标志性事件 |
| 自我状态 / 公开形象 | 自我状态层 | 现实人物的独有公开形象指纹 |
| 内在冲突 | 内在冲突层 | 是否在复刻现实人物的特定争议叙事 |

---

## 6. 反复修改的累积风险

即使用户的**每一次单独修改**都通过审核，也要警惕**累积可识别性**：

- 多次“安全替代”叠加后，是否整体重新指向了某现实人物？
- 用户是否在用一连串小修改，逐步逼近某个现实人物？

若发现累积后整体可识别，应：

1. 暂停并提示用户“当前多项设定的组合已开始指向某现实政治人物”；
2. 对整体 persona 重跑一次 `recognizability_review.md` 全流程；
3. 必要时回退部分修改，或整体重新转化为安全原型。

---

## 7. 速查清单（每次修改后自检）

- [ ] 本次修改不含近现代现实人物姓名 / 现实政党派系选举 / 政策口号 / 丑闻 / 家庭任职轨迹 / 遇刺审判政变下台事件？
- [ ] 不安全项已拒绝，并给出抽象的安全替代？
- [ ] 修改后的 persona 各层仍自洽（一致性检查通过）？
- [ ] 累积修改未让整体重新指向某现实人物？
- [ ] `meta.safety_status` 仍为 `PASS` / `safe_conversion`？
